# 厨房库存 Agent 原型技术方案

## 1. 文档目的

本文档描述一个以 **Python + LangChain** 实现的厨房库存 Agent 后端原型。原型的重点是验证 Agent 能否根据用户输入、食材识别结果、置信度、过敏原和用户记忆，自主选择并调用正确的业务工具。

本阶段提供一个轻量展示前端，但不追求生产级完整实现。复杂但暂时不实现的能力会保留在架构中，并在对应位置标记为 **【暂不实现】**。

## 2. 原型目标

原型需要证明以下能力：

1. Agent 能接收文本、食材、置信度、过敏原和用户记忆组成的上下文。
2. Agent 能在当前实现的库存管理和食谱搜索工具中自主选择；订单管理仅保留为完整方案中的扩展类别。
3. Agent 可以根据工具执行结果继续调用工具或生成最终回答。
4. Agent 不能绕过工具直接修改库存数据，也不能声称完成尚未实现的订单操作。
5. 库存新增、修改和删除必须生成待确认操作，只有白名单执行器能在用户确认后写入数据库。
6. 对食谱相关结果执行基础过敏原检查。
7. 从日志中可以观察 Agent 选择了什么工具、传入什么参数以及工具返回什么结果。

## 4. 总体流程

```text
文本 + 图片
    ↓
输入校验与简化鉴权
    ↓
规则意图识别
    ↓
调用图像识别 API：食材 + 置信度
    ↓
置信度分级处理
    ↓
读取过敏原、用户画像和历史
    ↓
组装 Agent Context
    ↓
LangChain Agent
    ↓
自主选择库存 / 食谱工具
    ↓
工具返回结果给 Agent，必要时继续调用
    ↓
基础输出审核与格式化
    ↓
最终结果
```

向量意图识别和完整三级审核属于完整方案的一部分，但不进入当前原型的实际执行链路。订单管理也只保留在总体设计中，不进入当前原型。

## 5. 技术选择

### 5.1 Python

Python 用于实现 Agent、业务工具、外部 API 适配器、数据模型和测试。

### 5.2 LangChain

LangChain 用于：

- 接入支持 Tool Calling 的大语言模型；
- 定义 Agent 的 System Prompt；
- 将普通 Python 业务函数包装为 Agent 工具；
- 管理 Agent、工具调用和工具结果之间的消息循环。

当前不使用 LangGraph。原型的流程只有一个 Agent 工具调用循环，不需要单独的图状态机。

如果未来真正实现人工确认、断点恢复、多级审核节点或复杂条件分支，再评估引入 LangGraph。

### 5.3 Pydantic

Pydantic 用于约束 Agent 上下文、工具参数和工具返回值，减少模型生成错误参数的风险。

## 6. Agent 设计

### 6.1 Agent 输入

Agent 不直接处理原始图片。图片服务的输出会先转化为结构化食材信息，再与其他信息一起交给 Agent。

Agent Context 包含：

| 字段 | 说明 |
|---|---|
| `user_id` | 当前测试用户标识 |
| `request_text` | 用户原始文本 |
| `intent` | 规则识别出的意图提示，不作为强制结论 |
| `ingredients` | 食材名称、置信度及置信度状态 |
| `allergens` | 用户已知过敏原 |
| `profile` | 饮食偏好等用户画像 |
| `history` | 少量历史摘要 |
| `tool_history` | 当前请求中的工具调用记录 |

### 6.2 Agent System Prompt

System Prompt 至少定义以下规则：

1. Agent 身份是厨房库存与食谱助手。
2. Agent 只能通过提供的工具查询或修改业务数据。
3. 不得声称未实际执行的库存、购物车或订单操作已经完成。
4. 食谱推荐必须考虑过敏原和已确认食材。
5. 中等置信度食材不得被当成确定事实，应在回答中要求确认。
6. 低置信度食材不得用于自动修改库存或确定性食谱推荐。
7. 信息不足时应说明缺少的信息，而不是编造工具结果。
8. 遇到禁止或恶意操作时拒绝执行。

完整恶意输入防御与 Prompt Injection 防护为 **【暂不完整实现】**，原型只保留基础禁止规则。

### 6.3 Agent 执行循环

```text
构造消息与上下文
    ↓
调用模型
    ↓
模型是否请求工具？
    ├─ 否 → 生成最终回答
    └─ 是
         ↓
      校验工具名称和参数
         ↓
      执行 Python 工具函数
         ↓
      记录调用与返回值
         ↓
      将工具结果交回模型
         ↓
      进入下一轮
```

为防止模型无限调用工具，原型设置最大循环次数。超过限制后停止执行并返回可解释错误。

## 7. 工具设计

所有已实现工具的业务逻辑均实现为普通 Python 函数。LangChain 只负责将函数暴露给 Agent，避免业务逻辑与框架强绑定。

### 7.1 库存管理工具

原型工具：

- `list_inventory()`：查询全部库存；
- `get_inventory_item(name)`：查询指定食材；
- `propose_add_inventory_item(name, quantity, unit)`：提出新增操作；
- `propose_update_inventory_item(name, quantity, unit)`：提出修改操作；
- `propose_remove_inventory_item(name, quantity)`：提出减少或删除操作。

三个写工具只创建 `pending_actions` 记录，不直接修改库存。确认接口使用 `inventory.add`、`inventory.update`、`inventory.remove` 白名单重新校验参数，并在确认后调用库存仓库；取消、过期、跨用户和重复确认均受到状态检查。

### 7.2 订单管理工具

订单管理属于完整方案中的 Agent 工具类别，当前原型 **【暂不实现】**，不提供购物车、下单、支付或配送工具。System Prompt 中应明确告知 Agent 当前不具备订单管理能力，避免模型声称已经完成相关操作。

### 7.3 食谱搜索工具

原型提供统一工具：

- `search_recipes(ingredients, preferences, allergens)`。

该工具调用已有食谱 API 返回候选结果，并执行基础过敏原过滤。

完整方案中的三级搜索策略保留如下：

```text
菜谱 API → RAG → Web Search
```

原型接入已有菜谱 API。RAG 和 Web Search 为 **【暂不实现】**，只在完整方案中作为后续降级检索层保留。

## 8. 意图处理

完整意图识别方案由两部分组成：

```text
简单匹配 → 向量意图识别
```

原型只实现非常简单的文本匹配作为占位，用少量关键词完成前置分流：

- 简单问候或帮助请求可以直接返回；
- 明确禁止的操作直接拒绝；
- 库存和食谱相关请求交给 Agent；
- 订单请求返回“当前原型暂不支持”；
- 无法确定的请求仍可交给 Agent 判断。

规则识别结果作为 Agent 的提示信息，而不是完全替代 Agent 决策。

向量意图识别为 **【暂不实现】**。当前简单匹配只用于证明数据能够进入 Agent，不以意图识别准确率为原型目标。

## 9. 图像识别与置信度

### 9.1 输入校验

原型可以实现：

- 文件是否存在；
- 文件大小是否超限；
- 文件扩展名与实际格式是否允许；
- 图片能否被正常读取。

### 9.2 图像识别

原型通过 API 调用已有图像识别服务，服务返回：

```text
食材名称 + 置信度
```

### 9.3 两个置信度阈值

设高阈值为 `T_high`，低阈值为 `T_low`：

| 区间 | 处理方式 |
|---|---|
| `confidence >= T_high` | 作为高可信候选交给 Agent |
| `T_low <= confidence < T_high` | 标记为需要用户确认 |
| `confidence < T_low` | 不作为有效食材参与自动操作 |

阈值在配置中定义，避免写死在 Agent Prompt 中。

## 10. 用户画像、历史与过敏原

原型从 Markdown 用户画像文档中读取：

- 已知过敏原；
- 饮食偏好；
- 少量历史摘要。

Markdown 文档可同时保存简化的用户画像和历史摘要。Agent 只读取这些信息，不实现自动长期记忆写入。

向量化长期记忆、记忆检索、过期机制和隐私管理为 **【暂不实现】**。

过敏原在原型中至少进行两次基础检查：

1. 食谱工具返回候选菜谱前检查；
2. 最终输出前再次检查。

完整方案中的三处检测逻辑保留，但第三处及更完整的同义词、复合配料映射为 **【暂不完整实现】**。

## 11. 输出审核

完整方案的三级输出审核依次为：

1. **正则匹配**：匹配禁止内容、敏感模式和必要格式；
2. **工具调用审查**：检查工具名称、参数、调用记录及最终陈述是否一致；
3. **另一个模型二次判断**：使用独立模型判断输出是否安全、合规且符合限制。

当前原型只实现非常简单的正则匹配占位。工具调用审查和另一个模型的二次判断均为 **【暂不实现】**，但保留对应接口位置。

## 12. 数据与状态

原型的数据来源如下：

- 用户画像和历史：Markdown 文档；
- 库存：简单数据库；
- 食谱：已有食谱 API；
- 食材识别：已有图像识别 API。

订单数据不保存。数据库只需满足库存基本增删改查，事务、并发控制和生产级幂等机制为 **【暂不实现】**。

每次请求应生成独立的请求标识，并记录当前请求内的工具调用历史，便于观察 Agent 行为。

## 14. 原型演示场景

至少准备以下场景：

1. 用户查询当前库存，Agent 调用库存查询工具。
2. 用户要求把高置信度识别出的食材加入库存，Agent 调用库存新增工具。
3. 用户根据现有食材请求菜谱，Agent 调用已有食谱 API 工具并考虑过敏原。
4. 图片中存在中等置信度食材，Agent 不将其视为确定食材，并要求确认。
5. 推荐结果涉及已知过敏原，工具阻止结果直接输出。
6. 用户请求订单管理时，Agent 明确说明当前原型不支持。
7. 简单正则规则命中禁止内容时，输出审核占位模块拒绝输出。

## 15. 原型完成标准

满足以下条件即可视为原型完成：

- 不依赖前端，可以通过命令行、Python 调用或测试运行；
- Agent 能在库存与食谱两类已实现能力中选择正确工具；
- 工具调用参数能够通过结构化校验；
- Agent 能使用工具结果生成最终回答；
- 置信度状态能够影响 Agent 行为；
- 过敏原能够影响食谱结果；
- 每次工具调用均有可查看的日志；
- 已实现与暂不实现内容在代码和文档中有明确标记。

## 16. 后续扩展位置

以下能力只预留位置，不属于当前原型工作：

1. 在简单匹配后增加向量意图识别；
2. 在已有食谱 API 后增加 RAG 和 Web Search；
3. 实现工具调用审查；
4. 接入另一个模型完成输出二次判断；
5. 实现订单管理工具；
6. 增加长期记忆与向量检索；
7. 需要复杂流程时引入 LangGraph；
8. 完善数据库事务、幂等、生产鉴权和监控。

这些扩展应通过替换适配器或增加节点完成，不应修改库存和食谱工具的核心业务函数。
